# セッション経緯の保存（auto memory への要約・追記）

会話セッションの末尾でユーザーが経緯保存を求めたときに起動する。今のセッションで行なった作業のうち、**将来の会話で価値のある非自明な事項**を抽出し、ユーザー auto memory に整理して書き込む。

## 起動条件

- 「ここまでの経緯を保存」「session を memory に残す」「ここまでをまとめて memory に書いて」
- 大きな作業の完了後にユーザーが明示的に保存を求めたとき
- `commit-push` 等で区切りがついたあと、ユーザーが続けて経緯保存を依頼したとき

## 絶対の制約

- **コードや git log から導ける情報は保存しない**: ファイル・パス、関数名、最近の変更、誰が何をいつ変えたかは git/code が示すので memory に重複しない
- **エフェメラルな会話状態は保存しない**: 進行中のタスクや一時的な決定は plan / TodoWrite に置き、memory には残さない
- **CLAUDE.md に既出の規約は再記載しない**: CLAUDE.md は別レイヤーで読み込まれる
- **MEMORY.md は索引、内容は別ファイル**: 1 行 ~150 文字以内、`[Title](file.md) — one-line hook` 形式
- **既存メモリと重複しない**: 既存があれば更新、なければ新規作成。重複エントリは作らない
- **分類を意識**: `user`（ユーザー像）、`feedback`（指示・好み・採用判断）、`project`（プロジェクト固有事実）、`reference`（外部リソースのポインタ）

## 保存に値するもの／しないもの

**保存する**:
- 非自明な決定: ユーザーが選んだ方針（A/B/C のうち A を採用、その理由）
- 指示・好み: 恒常的に適用される表記・用語・ルール（「○○ は△△と書く」「××を避ける」）
- 採用された判断: 提案のうちどれを採り、なぜか
- プロジェクト状態の更新: ラウンド回数、装飾規約、引用キー履歴、検証カウント等
- 方針上の学び: 今回の経験から将来も適用したい教訓
- 将来の参照先: 外部リソース、関連スクリプト、関連メモリのポインタ

**保存しない**:
- 個別の line number、ファイル名、関数名（コードを読めば分かる）
- 個別のコミット SHA・ファイル差分（git log で分かる）
- セッション中の議題そのもの、進行中の TODO、検証中の仮説
- 一過性のデバッグ過程

判断基準: 「次のセッションが今回と同じ作業を繰り返さないために何を覚えておくべきか?」を自問する。コードや git で再構築できる情報は対象外。

## 手順

### Step 1. メモリ・ディレクトリの確認

1. cwd に対応する memory ディレクトリを特定:
   - `~/.claude/projects/<project-slug>/memory/`
   - `<project-slug>` は cwd を `-` 区切りで slug 化したもの（例: `/path/to/my-project` → `-path-to-my-project`）
2. `MEMORY.md` を Read して既存の索引を把握
3. 関連しそうな既存メモリ・ファイル（`proofreading_history.md`, `feedback_*.md`, `project_*.md` 等）の冒頭または該当部分を Read

### Step 2. 抽出すべき事項のリスト化

セッションを振り返り、以下を箇条書きで列挙:

- (a) 今セッションで実行した一連の作業（git log と差分から要約。詳細はコード・git に残るので、ここでは「何を / なぜ」のみ）
- (b) ユーザーが採用した方針判断（提案の中からどれを選んだか + 理由）
- (c) 指示・好み（恒常的なルール・表記・用語）
- (d) プロジェクト状態の数値更新（ラウンド数、ファイル数、引用整合カウント等）
- (e) 方針上の学び（次回も適用したい教訓）
- (f) 将来の参照先（このセッションで言及された外部リソース）

各項目について、「これは既存メモリで覆われているか?」を判定:
- 既存メモリに該当エントリあり → 該当ファイルに追記または更新
- 該当エントリなし → 新規ファイル作成
- メモリ価値が低い（コード/git から導ける、エフェメラル）→ 保存しない

### Step 3. 既存メモリへの追記

該当する既存ファイルがあれば、`Edit` で該当節または末尾に追記。代表的な追記先:

- `proofreading_history.md`: 校正・改修履歴（プロジェクト系メモリ）
- `feedback_notation.md`: 装飾規約や記号の使い方（フィードバック系）
- `project_overview.md`: プロジェクト全体状態
- `project_submission_plan.md`: 投稿計画

履歴系ファイルへの追記は日付見出し（`### YYYY-MM-DD セッション（テーマ）`）で区切ると後で参照しやすい。

### Step 4. 新規メモリ・ファイルの作成

新規エントリが必要な場合:

1. ファイル名: `<type>_<topic>.md`（例: `feedback_naming_convention.md`、`project_round_18_notes.md`）
2. ファイル本体は次のフロントマター付き:

```markdown
---
name: {{タイトル}}
description: {{1 行説明（MEMORY.md の hook と一致させる）}}
type: {{user, feedback, project, reference}}
---

{{本文}}
```

3. **feedback / project の本文構造**: 結論を先頭に、続けて `**Why:**`（理由・背景）と `**How to apply:**`（適用条件）を入れる。これは将来の判断時にエッジケースを判別するため
4. MEMORY.md に索引行を追加（適切なグルーピング順序を維持）

### Step 5. MEMORY.md インデックスの更新

1. 既存エントリの hook を更新（ラウンド数、最新マイルストーン等）
2. 新規ファイルがあれば索引行を追加
3. 1 行 ~150 文字以内、`- [Title](file.md) — one-line hook` 形式
4. 200 行を超えないよう注意（200 行以降は会話に注入されない）

### Step 6. ユーザーへの報告

書き込んだファイル一覧と、各ファイルで追加 / 更新した内容のサマリー（1〜2 文ずつ）を報告。

## 既存ファイルへの追記時の典型構造（履歴系）

```markdown
### YYYY-MM-DD セッション（テーマ）

コミット abc1234 → def5678 にかけて、{{作業種別}}を実施。

**コミット abc1234「件名」**:
- 変更点 1: {{何を変えた / なぜ}}
- 変更点 2: ...

**コミット def5678「件名」**:
- ...

**結果**: {{ラウンド結果、検証カウント、ページ数等の事実}}

### 方針上の学び（YYYY-MM-DD ラウンド）

- {{教訓 1}}: {{適用条件と理由}}
- {{教訓 2}}: ...
```

## 新規 feedback / project ファイルの典型構造

```markdown
---
name: {{ルール名・事実名}}
description: {{1 行説明}}
type: feedback
---

{{結論（ルールそのもの）}}

**Why**: {{背景・理由・過去の事例}}

**How to apply**: {{いつ・どこで適用するか、エッジケースの判断基準}}
```

## チェックリスト

- [ ] cwd の memory ディレクトリを特定したか（`~/.claude/projects/<slug>/memory/`）
- [ ] MEMORY.md と関連既存ファイルを Read したか
- [ ] 「保存に値する」事項のみを抽出したか（コード/git 由来は除外）
- [ ] 既存エントリへの追記 vs 新規ファイル作成を判定したか
- [ ] 新規ファイルにフロントマター（name, description, type）を付けたか
- [ ] feedback / project の本文に `**Why:**` と `**How to apply:**` を入れたか
- [ ] MEMORY.md インデックスを更新したか（hook の更新 / 新規行追加）
- [ ] MEMORY.md の総行数が 200 行以下か
- [ ] 重複エントリを作っていないか

## 関連スキル

- `commit-push` — セッションの変更をコミット・プッシュ。経緯保存はその直後に行なうのが自然
- 本スキルは「session 末尾の batch update」を担う。会話途中の per-fact な記憶は auto memory プロトコル（システム・プロンプトに既出）に従って即時保存する

## 設計ノート

- システム・プロンプトの auto memory は会話中の **per-fact 即時保存**を担う。本スキルは **session 末尾の batch summary** を補完する
- batch update が必要な理由: 大きな作業（例: 数学検証 1 ラウンド + 5 件の改修 + 学び 5 項目）は、per-fact では分散しすぎて文脈が失われる。1 つのコミット粒度でまとめることで、後の参照時に「あのときの session で何が決まったか」を辿りやすい
