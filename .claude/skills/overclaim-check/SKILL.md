# 論文の overclaim（言いすぎ）検証をバックグラウンドで実行

1 本または複数の `.tex` ファイルに対し、**要旨・はじめに・命題本体・系・結論・実証解釈で主張されている内容が、実際に証明・推定・観察された範囲を超えていないか**を徹底検証し、結果を `doc/overclaim_report_X.txt` に保存する。

役割分担：
- `proofread-paper`: 文章表現・表記揺れ・引用文献メタデータ・LaTeX 体裁
- `math-proof-check`: 証明の論理的厳密性・数学的整合性・数値計算の独立検証
- `citation-check`: 引用文献の中身と本文の主張が整合しているか
- **`overclaim-check`（本スキル）**: 「主張が、結果として導かれた範囲を超えていないか」という外延的妥当性

実行は**バックグラウンドの Agent** に委譲する。スキル本体（メイン会話）はファイル番号の決定・プロジェクト設定の読込み・Agent 起動のみを行なう。

## 引数

- `$ARGUMENTS`（省略可）: 検証対象の `.tex` ファイル（スペース区切りで複数指定可、相対または絶対パス）。
- 省略時は **`<cwd>/.claude/proofread.md`** をプロジェクト設定として読み込み、そこに列挙された対象ファイルを使用する（他校正系スキルと共通）。
- `.claude/proofread.md` も存在しない場合はユーザーに対象ファイルを確認する。

## 絶対の制約

- **対象の .tex ファイルは絶対に直接編集しない**。検証結果は `doc/overclaim_report_X.txt` に保存するのみ。
- LaTeX のビルド・コミット・プッシュもしない。
- バックグラウンドで実行する（メイン会話をブロックしない）。
- 修正・ビルド・コミット・プッシュは、レポート確認後にユーザーが明示的に指示した場合のみ別ステップとして実行する。
- 時間がかかってもよいので徹底することを優先する。

## 手順

### Step 1. プロジェクト設定の読込み

1. `$ARGUMENTS` に対象ファイルの明示があればそれを使う。
2. なければ `<cwd>/.claude/proofread.md` を Read する。
   - 存在すれば、そこに記載された対象ファイル・プロジェクト固有の既知数値・追加ルールを取得。
   - 存在しなければユーザーに対象ファイルを確認。

### Step 2. 出力ファイル番号 X の決定

- `<cwd>/doc/overclaim_report_<n>.txt`（n は自然数）を Glob で列挙
- 未使用の**最小の**自然数 X を決定
- `doc/` ディレクトリが存在しない場合は Agent 側で作成する

### Step 3. 付属ファイルの自動検出

対象 `.tex` と同じディレクトリおよびプロジェクト直下を調査：

- **`.bib` ファイル**があれば、引用文献の主張範囲との対比に利用
- **数値検証スクリプト**（`bin/verify_*.jl`, `bin/*_check.wlnb` 等）があれば、本文の数値主張の限定範囲確認に利用
- **過去の `math_proof_check_*.txt` / `proof_reading_*.txt` / `citation_check_*.txt`**：すでに指摘済みの限定条件（例：パラメタ領域の限定、Remark 化された比較静学の符号判定限定など）を把握し、本文の主張がそれと整合しているか確認

### Step 4. バックグラウンド Agent の起動

`Agent` ツールを以下のパラメータで呼び出す：

- `subagent_type: general-purpose`
- `description: "Thorough overclaim check round X"`
- `run_in_background: true`
- `prompt`: 下記「Agent への指示テンプレート」に、以下を埋め込んで渡す：
  - `X`（ラウンド番号）
  - 対象 `.tex` ファイルの絶対パス一覧
  - `.bib` ファイルの絶対パス（検出できた場合）
  - 数値検証スクリプトの絶対パス（検出できた場合）
  - `<cwd>/.claude/proofread.md` の内容（プロジェクト固有の既知数値・追加ルール）
  - 出力先の絶対パス `<cwd>/doc/overclaim_report_X.txt`

### Step 5. ユーザーへの通知

Agent 起動後、即座に報告：

- バックグラウンドで第 X 回 overclaim 検証を開始したこと
- 出力先（絶対パス）
- 完了時に通知される旨

Agent の進捗を polling しない。

### Step 6. 連続ゼロ件への到達判断（任意）

ユーザーが「問題ゼロの回が N 回続くまで繰り返す」と指示した場合は、完了通知を受けて以下を判定：

- 直近 N 回のレポートがすべて Critical 0 / Major 0 / Minor 0 / Style 0 ならば終了
- 1 件でも検出があれば、ユーザーに修正プランを提示 → 修正後に本スキルを再起動
- 次回の Agent には「前回と異なる独立視点で検証せよ」と明示指示することで検証の多様性を確保（同じ overclaim パターンを繰り返さない）

【検証観点】

## A. 要旨・はじめに・おわりにでの強さ
- 「証明された」「示した」「決定的に」「明らかに」「すべて」「常に」「ロバスト」「一般に」など強い断定の使用
- モデル特定パラメタ領域（例：$C<0$, $\Phi^*<1$, 内点性 $\beta_t^*\in(0,1)$ 等）でのみ成立する結果を、その限定を伏せて一般主張として書いていないか
- 命題の限定条件（仮定 ass1, ass2 等）が要旨・はじめに・おわりにから抜けていないか
- 数値計算で確認しただけのことを「証明」と書いていないか
- 「投資の質と裁量的アクルーアル」のような「正の関連」を、「シグナリングを実証した」「市場が情報として受容することを示した」と即断していないか（市場反応分析が論文の対象外なら、その留保を明示しているか）

## B. 命題・系の主張範囲
- 命題本体の主張と証明が成立する範囲が一致しているか（証明より広い結論を述べていないか）
- 「一意」「無条件」「すべての」など強い量化子の使用が、本当に正しいか（部分的にしか証明していない可能性）
- 主仮定外でも成立すると示唆していないか
- 狭義不等号・広義不等号の使い分けが、証明された範囲と一致しているか
- 「いつでも」「常に」が、内点性などの暗黙の仮定で限定されているのに、その限定が命題に明示されていないか

## C. 実証結果の解釈
- 統計的有意性のみで経済的含意を語る overclaim（係数の大きさが現実的に意味を持つか）
- サンプル特性（地域・期間・産業）の限定範囲を超えた一般主張（「国際企業」と「16 市場」の代表性、特定年代の経済環境の限定）
- 内生性・選択バイアスを十分にコントロールできていない時の因果主張（「causes」「effects」「signals」「leads to」など）
- 相関の方向性主張（「正の関連」）を超えて、強い因果（「決定する」「説明する」「説明しきる」）を主張していないか
- 平均的な相関を、個別企業レベルの予測力に拡張していないか

## D. 因果と相関の区別
- パネル回帰の係数有意性を「因果効果」と即断していないか
- 操作変数・自然実験・差分の差分のような同定戦略を使っていない場合、語彙を「関連」「予測」に弱化すべき箇所が「効果」「影響」になっていないか
- "explains" / "drives" / "determines" / "leads to" / "causes" / "signals" の使用範囲

## E. パラメタ領域の限定
- 命題が特定パラメタ領域（例：$C<0$, $\Phi^*<1$, $\partial F_2/\partial\mu>0$）でのみ成立する場合、その限定が要旨・実証仮説・結論で適切に伝達されているか
- 数値ベンチマーク 1 つで確認した結論が、命題の主張として一般化されていないか
- 「典型的なパラメタ領域」「現実的範囲」のような曖昧表現で限定をぼかしていないか

## F. シグナリングと情報伝達の主張
- モデルが導出するのは「正の関連性」だけで、シグナリングという用語の使用が市場反応・受け手の解釈・均衡選択など追加前提を含むなら、その前提が明示されているか
- 「シグナリング機能を実証した」と書きながら市場反応を検証していない場合、その留保が明示されているか
- 「conditioning signaling」「signaling channel」のような用語の使用が、実証パートで実際に同定されたものと整合しているか

## G. 関連文献との位置取り
- 「既存研究と一線を画す」「初めて示した」「これまでなかった」のような新規性主張が事実か（先行研究の探索範囲は十分か）
- 「先行研究は X を示さなかった」のような否定的主張が、引用文献に対する適切な要約になっているか（citation-check 系の問題と被るが、overclaim 視点では「先行研究を低く位置付けすぎていないか」を見る）
- 既存モデルの一般化・特殊化の関係が誤っていないか（「既存研究を含む」と書きながら実は特殊化になっている等）

## H. 数値例の取り扱い
- 推奨ベンチマーク数値を、命題の「代表例」と書いている場合、その範囲が他のパラメタ領域に一般化できると誤読されないか
- 「数値計算で確認した」「数値例で支持された」が「証明された」「成立する」に格上げされていないか
- supremum/infimum の数値（例：$\Phi^*$ の supremum ≈ 0.92）が、本文中で「上限」「最大」と表現されている場合、それが達成可能か否かの区別がついているか

## I. 用語の重みと整合性
- 「decisive」「dominant」「robust」「general」「universal」「fundamental」のような重い形容詞の使用
- 単純な十分条件を「特徴づけ」（characterization）と呼んでいないか
- 「内点最適化」「最適契約」と書きながら一階条件のみで二階条件の確認に触れていないか（最適性主張）

## J. Discussion・Conclusion の主張範囲
- 結論部での「政策的含意」「規制への示唆」が、論文の知見から導かれる範囲を超えていないか
- 「実務への含意」が、推定された相関から直接導けない量の主張になっていないか
- 「今後の研究」と書きながら、本論文がすでに解決したかのような表現が混ざっていないか

【重大度分類】
- **Critical**: 主張範囲が完全に誤っており、根本的な書き換えが必要（例：因果関係を主張しているが同定戦略がない、証明の限定領域外まで主張している、論文の核心結論が成立しない）
- **Major**: 主張範囲が明らかに広すぎ、限定条件の明示か語彙の弱化が必要（例：要旨に限定が抜けている、シグナリングという用語の範囲が広すぎる）
- **Minor**: 限定条件の省略、軽微な overclaim、語句の重さが過剰
- **Style**: 強い形容詞の使用、表現の控えめ化推奨

【手順】
1. 対象 .tex をすべて全文 Read（長大なら offset/limit で複数回に分割）
2. .bib と数値検証スクリプトの存在確認、必要に応じて Read
3. 直近の overclaim_report_*.txt / math_proof_check_*.txt / proof_reading_*.txt / citation_check_*.txt、CLAUDE.md、memory を Read
4. 各セクションの主張を列挙し、命題・系・実証結果に対応付けて「主張範囲 > 結果範囲」になっていないかをマッピング
5. Grep で「強い断定の語彙」（「証明」「決定」「明らか」「ロバスト」「一般に」「無条件に」「常に」"always" "robust" "general" "decisive" "universal" "fundamental" "characterize" "causes" "implies" "demonstrates" "establishes" "proves" 等）を網羅検出し、各使用箇所が主張範囲として正しいか個別に検証
6. 要旨・はじめに・命題本体・系・実証仮説・結論を 1 つずつチェックし、限定条件の明示状況を確認
7. 命題が成立する限定領域（既知数値・既知の Remark から取得）と、本文の主張範囲のマッピングを作成
8. 検出した各 overclaim に対し、修正案（書き換え例）を提示
9. レポートを Write で出力ファイルに保存

【本ラウンドの独立視点（過去ラウンドとの差別化）】
過去の overclaim_check が既に実施済みの場合は、**別の切り口**で独立検証すること。たとえば：
- ラウンド N-1 が「要旨・はじめに中心」ならラウンド N は「命題本文の量化子・実証仮説の限定条件中心」
- ラウンド N-1 が「シグナリング用語の範囲」ならラウンド N は「因果語彙・パラメタ領域限定の明示」
- ラウンド N-1 が「強い形容詞の語彙」ならラウンド N は「先行研究との位置取り・新規性主張」
同じ切り口を繰り返すだけでは検証の多様性が得られない。

## `.claude/proofread.md` の共通形式

プロジェクト固有情報は `<cwd>/.claude/proofread.md` に書き、proofread-paper / math-proof-check / citation-check / overclaim-check で共有する。

スキル側はこのファイルを無加工で Agent プロンプトに埋め込む。

## 関連スキル

- `proofread-paper`: 文章表現・引用文献体裁の校正。主張範囲は扱わない。
- `math-proof-check`: 証明の数学的厳密性。証明そのものの正しさを見る。
- `citation-check`: 引用先文献の内容と本文の主張の整合。
- **組み合わせ運用推奨**: 大改訂前は `math-proof-check` でゼロ件到達 → `citation-check` で引用整合 → `overclaim-check` で主張範囲縮約 → `proofread-paper` で仕上げ、の順が効率的。証明が正しくても主張範囲が広すぎる、引用が誤っているといった問題は別レイヤーで、別個に検証するのが実績のあるワークフロー。
